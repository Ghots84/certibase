# Spine Pair Review — CertiBase

## Overall verdict
The pair is mechanically sound where it's been exercised carefully — all 17 tokenized components have full DESIGN.md visual specs, all 4 mockups are linked cleanly with no orphans or dangling links, and the "spines win" rule is stated once per file, not belabored. The gaps cluster in two places: the Accessibility Floor audited one contrast pair (white-on-primary) and missed a materially worse one in the same neutral family (`text-faint`), and EXPERIENCE.md's behavioral coverage thins out for buttons and several display components even though DESIGN.md fully specifies them visually. Neither file is broken as a contract, but a downstream consumer would hit real ambiguity implementing buttons, the import validation card, and Assistant's state machine.

## 1. Flow coverage — adequate
Checked all 3 Key Flows (EXPERIENCE.md → Key Flows) for protagonist, numbered steps, bolded climax, and failure path.
### Findings
- **high** No flow has a CSM/Sales agent browsing Fiches directly (category grid → `fiche-card` → drawer) as the protagonist's primary action — the product's core "base de connaissance" surface, consulted "plusieurs fois par jour" per Brand & Style, is only touched tangentially in Flow 2 (an admin/sales validation action reached from Imports, not from Fiches nav). (EXPERIENCE.md → Key Flows) *Fix:* Add a 4th flow, or swap Flow 3's setup to originate from a Fiches search rather than the Assistant directly.
- **low** No flow touches Utilisateurs — defensible: it's admin-only, has no dedicated mockup by explicit decision (`.decision-log.md` → "Couverture des écrans"), and is the least product-differentiated surface (a member table). *Fix:* none required; note as accepted scope.
- No misses on structural mechanics: all 3 flows have a named protagonist, numbered steps, an explicit **Climax** beat, and an "Échec :" failure paragraph (EXPERIENCE.md lines 92-118).

## 2. Token completeness — thin
Extracted every frontmatter token (DESIGN.md) and every `{path.to.token}` reference in both files' prose; checked resolution and computed contrast for load-bearing text/background pairs.
### Findings
- **critical** `status-badge` "archived" variant (`dot`/`color`: `text-faint` `#8A94A2`, `background`: `surface-2` `#F1F3F6`) computes to ≈**2.8:1** contrast — fails AA even at the large-text/UI-component floor (3:1) — and is completely undocumented in Accessibility Floor, which only audits white-on-`primary`. (DESIGN.md lines 234-238; EXPERIENCE.md → Accessibility Floor) *Fix:* Add this pair to the Accessibility Floor audit; either darken `text-faint` for this variant or use `text-muted`.
- **high** `text-faint` (`#8A94A2`) on `surface`/white (`#FFFFFF`) computes to ≈**3.1:1** — below AA normal/small text (4.5:1). This pairing is used by `stat-card.sub-color`, `fiche-card.hint-color`, `alert-item.sub-color`, and `empty-state.plain-color` (DESIGN.md lines 154, 193, 296, 365) — none of these are mentioned in Accessibility Floor, which audits only one color pair. *Fix:* Extend Accessibility Floor with this combination, or accept-with-rationale as was done for primary-on-white.
- **medium** `{components.empty-state.tinted}` (EXPERIENCE.md → State Patterns, "Flux calme" row) doesn't resolve to an exact frontmatter key — only `tinted-background`, `tinted-border`, `tinted-radius`, `tinted-color` exist. DESIGN.md self-consistently uses wildcard notation `{components.empty-state.tinted-*}` for the same concept (Do's/Don'ts table, line 471); EXPERIENCE.md drops the wildcard and reads as an exact-but-missing key. *Fix:* Align notation — either `{components.empty-state.tinted-*}` in EXPERIENCE.md or spell out the specific sub-token actually meant.
- Confirmed: the documented white-on-`primary` contrast figure (~3.3:1, EXPERIENCE.md → Accessibility Floor) is numerically correct (computed independently at 3.325:1) and no color token is missing a hex/value in the frontmatter.

## 3. Component coverage — thin
Extracted all 17 `components.*` frontmatter keys plus every component named only in prose; checked for a DESIGN.md visual row and an EXPERIENCE.md Component Patterns row with real behavioral rules.
### Findings
- **high** `button-primary` / `button-secondary` have full DESIGN.md visual specs (disabled state, danger-hover) but no EXPERIENCE.md Component Patterns row anywhere — no stated rule for when a primary CTA is disabled, whether it shows a loading state during the approve/reject cycle described in the Imports row, or how `button-secondary`'s "hover-color-danger" ("Archiver") is triggered/confirmed. Buttons are the most universal interactive primitive in the system. (DESIGN.md lines 321-341; absent from EXPERIENCE.md → Component Patterns) *Fix:* Add a `button-primary`/`button-secondary` row to Component Patterns.
- **medium** (reverse gap) EXPERIENCE.md's "Carte de validation (import)" row (Component Patterns, Imports) specifies real behavior (optimistic `pending`→`processing`→`approved`/`rejected`, 0.45 opacity on reject) but has no matching DESIGN.md visual component — DESIGN.md mentions it only in passing inside the `confidence-bar` entry ("carte de validation d'import" as a third usage location) without defining its own background/border/radius/layout. This surface is load-bearing for Flow 2. (EXPERIENCE.md line 57; DESIGN.md line 451) *Fix:* Give it its own DESIGN.md components entry (likely inheriting `fiche-card`'s shape).
- **medium** `stat-card`'s value-color logic ("succès si nombre normal, `text-faint` si zéro/neutre, `warning`/`accent` si actionnable") is a behavioral rule embedded in the visual spec (DESIGN.md line 445) with no EXPERIENCE.md counterpart. *Fix:* Either move/mirror this rule into Component Patterns or accept DESIGN.md as its home and cross-reference.
- **medium** `confidence-bar` is described as "un seul composant, trois emplacements" (DESIGN.md line 451) but has no dedicated Component Patterns row; only partially surfaced via the "Confiance basse sur un draft" State Patterns row (low-confidence badge trigger), leaving its behavior on the other two surfaces (fiche-card, fiche-drawer) unstated. *Fix:* Add a one-line Component Patterns row, or note explicitly that it's static/display-only everywhere.
- **low** `type-badge`, `status-badge`, `shortcut-card`, `empty-state` have no Component Patterns row but are reasonably covered elsewhere — double-encoding rule for badges (Accessibility Floor), hover-lift rule for `shortcut-card` (Interaction Primitives), and full variant/trigger table for `empty-state` (State Patterns). Defensible given they're largely non-interactive/display components.
- DESIGN.md side is complete: all 17 frontmatter components have a prose row in DESIGN.md → Components (no misses).

## 4. State coverage — adequate
Walked each IA surface (Vue d'ensemble, Fiches, Concurrents, Imports, Assistant, Utilisateurs) against plausible states (empty, cold-load, error, permission-denied, draft/pending) and checked State Patterns + surrounding sections.
### Findings
- **high** Assistant has no row in the systematic State Patterns table at all — the "Chargement à froid" row explicitly lists "Fiches, Concurrents, Imports" and excludes Assistant — even though it clearly has loading (animated dots), streaming, and SSE-error states. These are only narrated informally inside Key Flow 3 ("Erreur : {message}"), not captured as a reusable state contract a downstream consumer could implement without reading a flow narrative. (EXPERIENCE.md → State Patterns, line 63; contrast with Component Patterns line 53 and Key Flow 3 line 118) *Fix:* Add an Assistant row (or rows) to State Patterns covering loading/streaming/error explicitly.
- **medium** Utilisateurs has zero state coverage anywhere (no State Patterns row, no Component Patterns row) despite managing a real invite/role workflow, not just a static list — no stated empty-org, invite-pending, or error state. Given the decision log's explicit "no dedicated mockup" call, thin visual coverage is expected, but the total absence of *any* state definition for an actual workflow (not just a table) is a real content gap, not just a mockup gap. *Fix:* At minimum, state the invite-pending lifecycle and an empty-org case.
- **medium** Imports has no explicit "no imports yet" first-use empty state — the empty-state rows in State Patterns are scoped to Fiches only ("Liste vide (jamais alimentée)", "Liste vide (filtre sans résultat)"); Imports only gets pipeline-in-progress, low-confidence, and generic network-error rows. *Fix:* Add an Imports empty-state row.
- **low** Vue d'ensemble has no explicit error-state entry (SSR failure mode unstated) — likely low-stakes given it's server-rendered and the zero-value stat-card color rule partially substitutes, but it's not named.
- Fiches and Concurrents are strong: cold-load, dual empty-state nuance, network error, permission pattern (admin-only category), and full draft→published→archived lifecycle are all named with clear triggers and copy.

- "Spines win on conflict" is stated exactly twice total: once in DESIGN.md → Components ("ce texte fait foi en cas d'écart") and once in EXPERIENCE.md → Information Architecture ("la spine gagne toujours") — one instance per file, not repeated redundantly within either file. Acceptable.

## 6. Bloat & overspecification — adequate
### Findings
- **medium** Migration Notes (EXPERIENCE.md, invented section) names concrete implementation identifiers — `app/globals.css`, `ROLE_COLORS`/`ROLE_LABELS`, `MAIN_NAV`, `getStats()`, specific paths like `fiches/page.tsx`/`imports/page.tsx`/`sidebar.tsx`/`users/page.tsx`. This is genuinely useful sequencing information, but this level of code-identifier specificity is architecture-doc territory more than spine territory — it will go stale on the next refactor independent of any design decision, and duplicates what an architecture.md handoff would normally own. *Fix:* Keep the 5-pass sequencing logic (that's legitimately UX-spine-owned, since it's about visual-risk ordering) but move specific code identifiers to an architecture note or footnote them as "current-state pointers, verify before use."
- No pixel specs found where a token already covers the same value; no restated narrative disconnected from a decision (e.g., the "h1 sizes vary 17/18/24px today" framing directly motivates the `heading-lg` unification Do/Don't). Overall the pair is lean for its component count.

## 7. Inheritance discipline — strong
### Findings
- Component names are identical across both files everywhere both exist (`sidebar-nav`, `fiche-card`, `category-card`, `chat-bubble`, `price-compare`, `comparison-card`, `alert-item`); `drawer`/`modal` are explicitly aliased via `fiche-drawer (`{components.drawer}`)` and `fiche-form-modal (`{components.modal}`)` rather than silently renamed. No camelCase/PascalCase naming drift found (grepped for `ficheCard`, `FicheCard`, `statCard`, `typeBadge`, `statusBadge` variants — none present).
- All `{path.to.token}` references resolve to real DESIGN.md keys with one exception carried from §2: `{components.empty-state.tinted}` (EXPERIENCE.md) vs. the wildcarded `{components.empty-state.tinted-*}` (DESIGN.md).

## 8. Shape fit — strong
### Findings
- DESIGN.md section order matches the canonical order exactly: Brand & Style → Colors → Typography → Layout & Spacing → Elevation & Depth → Shapes → Components → Do's and Don'ts.
- EXPERIENCE.md has all required sections present, in order: Foundation, Information Architecture, Voice and Tone, Component Patterns, State Patterns, Interaction Primitives, Accessibility Floor, Key Flows.
- The invented "Migration Notes" section (appended after Key Flows) earns its place — it's product-specific (real 5-pass sequencing tied to the decision log's explicit migration mandate, not generic advice) rather than filler. Per §6, its code-identifier granularity is a scope-boundary concern with architecture docs, but the section itself is justified for a brownfield-migration UX spine.
