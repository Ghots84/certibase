# Decision Log — CertiBase PRD

> Ce log documente les décisions structurantes prises lors de la rédaction du PRD CertiBase v1.0 (draft).
> Chaque décision inclut sa justification et son impact produit/technique.

---

## DEC-001 — Scope MVP : usage interne uniquement

**Date :** 2026-06-02
**Décision :** CertiBase est déployé en usage interne CertiPlace uniquement pour le MVP. Pas de multi-tenant déployé aux clients organismes certificateurs pour l'instant.
**Raison :** Simplification radicale du scope : pas de gestion multi-organismes, pas d'isolation des données par tenant, pas d'UI de configuration par client. Permet de livrer de la valeur interne rapidement et de valider les usages avant de complexifier.
**Impact :**
- `tenant_id` présent dans le schéma (colonne nullable) mais non exposé dans l'UI MVP
- RLS simplifié : authentification = accès aux données de l'organisation CertiPlace
- Les agents externes (CSM Digital, Sales Digital) sont développés mais servent la validation interne avant tout déploiement client
- La roadmap multi-tenant est reportée post-MVP

---

## DEC-002 — Simplification des rôles et accès

**Date :** 2026-06-02
**Décision :** 4 rôles uniquement : `admin`, `csm`, `sales`, `new`. Pas de rôles par tenant, pas de permissions granulaires par fiche ou module.
**Raison :** Cohérence avec la décision DEC-001 (usage interne). La complexité RBAC fine est injustifiée pour une équipe interne de taille limitée. Le rôle `new` permet d'onboarder sans assigner immédiatement.
**Impact :**
- RLS Supabase basé sur 4 règles simples (lecture vs écriture, admin vs other)
- L'assistant interne adapte ses réponses via `system prompt` différencié par rôle (profil-aware)
- Pas de UI de gestion des permissions — uniquement invitation + assignation de rôle par l'admin

---

## DEC-003 — IA interne promue Must (v1.1 vs v1.0)

**Date :** 2026-06-02
**Décision :** L'assistant IA interne (FR-17 à FR-22) est classé **Must** dans la priorisation MoSCoW, livré en S5–S6.
**Raison :** Le Brief v1.1 a identifié P4 (équipes sans assistant IA) comme un problème métier direct à résoudre. L'impact sur la productivité des CSM, Sales et Ops est immédiat. Techniquement, le coût d'implémentation est minimal (réutilisation de la route `/api/chat` existante, différenciation par system prompt).
**Impact :**
- Sophie (persona Ops) entre dans le scope MVP
- Metric "−60 % temps de recherche interne" devient mesurable dès S6
- Dépendance S5 sur la qualité de la base de fiches (besoin ≥ 50 fiches publiées pour que l'assistant soit utile)

## DEC-004 — Fast Path choisi pour la rédaction du PRD

**Date :** 2026-06-02
**Décision :** Le PRD est rédigé en mode Fast Path (draft avec assumptions) sans phase d'élicitation interactive.
**Raison :** Les sources disponibles (CDC v1.0 + Brief v1.1) fournissent suffisamment de contexte pour produire un PRD directement utilisable. Les zones d'incertitude sont documentées via des tags `[ASSUMPTION]` dans le PRD et des questions ouvertes en section 8.
**Impact :**
- Le PRD est un draft à valider par les parties prenantes avant le démarrage des stories
- 7 questions ouvertes identifiées (section 8 du PRD) nécessitent une réponse avant les sprints concernés
- Les assumptions sont traçables — chaque `[ASSUMPTION]` peut être résolu lors d'une session de revue

## DEC-005 — Principe "1 appel Claude = 1 JSON"

**Date :** 2026-06-02
**Décision :** Le pipeline d'extraction produit toutes les fiches drafts d'un import en un seul appel Claude, retournant un JSON structuré avec toutes les catégories (objections, FAQ, moments clés, angles morts).
**Raison :** Simplicité d'implémentation, coût API maîtrisé, facilité de debugging. Une chaîne de prompts séquentiels serait plus fragile et plus coûteuse. Le modèle claude-sonnet-4-6 a la capacité de tenir la complexité en un seul appel.
**Impact :**
- Pipeline plus robuste (1 point de défaillance vs N)
- Idempotence simplifiée (relancer = 1 appel, pas une chaîne à reprendre au bon endroit)
- Limite : si le fichier source est très long, la fenêtre de contexte peut être atteinte — mitigation via chunking si nécessaire (reporté post-MVP si non déclenché)

---

## DEC-006 — Pas de PyMuPDF, traitement PDF natif par Claude

**Date :** 2026-06-02
**Décision :** Les PDFs sont traités directement par l'API Claude (vision/multimodal natif), sans dépendance Python ou librairie de parsing PDF tierce.
**Raison :** Simplification de la stack (100 % Node.js/TypeScript), pas de serveur Python à maintenir, pas de conteneurisation supplémentaire. Claude traite les PDFs nativement avec une qualité équivalente ou supérieure à PyMuPDF pour l'extraction de sens (vs extraction de texte brut).
**Impact :**
- Stack unifiée Next.js + Node.js — aucune dépendance Python
- Limite : pour les PDFs très volumineux (> 100 pages), un chunking page par page peut être nécessaire — traité comme edge case post-MVP

---

## DEC-007 — Index pgvector ivfflat obligatoire dès le setup

**Date :** 2026-06-02
**Décision :** L'index `ivfflat` sur la colonne `embedding` de la table `fiches` est créé lors de la migration initiale (S1–S2), pas ajouté a posteriori.
**Raison :** Sans index, les requêtes vectorielles dégradent exponentiellement avec le volume. Créer l'index dès le début évite une migration de production sur une table avec données. La cible NFR-01 (< 2s p95) n'est pas atteignable sans index au-delà de 1 000 fiches.
**Impact :**
- Inclus dans le script de migration S1–S2
- Paramètre `lists = 100` adapté à une base de 10 000 fiches max — à ajuster si le volume dépasse

---

## DEC-008 — Agents externes inclus dans le MVP (S7–S10)

**Date :** 2026-06-02
**Décision :** Le CSM Digital (S7–S8) et le Sales Digital (S9–S10) sont inclus dans le périmètre MVP à 10 semaines, bien que le déploiement client multi-tenant soit hors scope.
**Raison :** Ces agents représentent deux des trois métriques primaires de succès (−40 % tickets, leads 24h/24). Les exclure du MVP viderait le produit d'une grande partie de sa valeur annoncée. L'usage "interne uniquement" ne les empêche pas — le CSM Digital peut être testé par l'équipe CertiPlace elle-même, le Sales Digital sur une version staging de certiplace.fr.
**Impact :**
- La dépendance sur l'API contexte client CertiPlace (Q2) doit être résolue avant S6
- La méthode d'intégration du widget (Q1) doit être validée avant S5
- Le déploiement en production client est une étape post-MVP, pas dans le scope de ce PRD

---

## DEC-009 — Seuil de confiance RAG à 0,75 par défaut

**Date :** 2026-06-02
**Décision :** Le seuil de similarité cosinus pour le RAG est fixé à 0,75 par défaut, configurable par admin.
**Raison :** Valeur empirique standard pour les applications RAG avec embeddings `text-embedding-3-small`. Trop bas → hallucinations par contexte hors sujet. Trop haut → taux d'escalade excessif en phase de base vide. 0,75 est le bon équilibre de départ.
**Impact :**
- NFR-04 (escalade sur faible confiance) s'appuie sur ce seuil
- La contre-métrique "taux d'escalade > 50 %" servira à ajuster ce seuil en production
- Q3 (seuil par type de fiche) reste ouverte — implémentation dans le schéma (`confidence_threshold` par fiche) mais UI de configuration reportée

## DEC-010 — Spike de validation ajouté en S0

**Date :** 2026-06-06
**Décision :** Une semaine de spike (S0) est ajoutée avant le build de l'infrastructure. L'équipe crée 10-15 fiches manuellement, monte un RAG minimal, et le teste avec 3 utilisateurs internes (Claire, Marc, Sophie).
**Raison :** L'hypothèse centrale "fiches structurées + RAG = meilleures réponses IA" n'était pas validée. Un spike de 3-5 jours peut éviter 10 semaines d'investissement sur une mauvaise structure de fiches.
**Impact :**
- Roadmap passe à 11 semaines (S0 + S1 à S10)
- FR-00 ajouté avec critères de succès mesurables
- Si le spike échoue, la structure des fiches est révisée avant S1

## DEC-011 — Décision conditionnelle A/B sur l'API CertiPlace (FR-12)

**Date :** 2026-06-06
**Décision :** FR-12 a deux scénarios documentés. Scénario B (sans API) est le plan par défaut ; Scénario A (avec API) est l'upgrade si confirmée disponible avant S5.

**Impact :**
- Décision A/B avant S5, planning S7-S8 basé sur Scénario B par défaut
